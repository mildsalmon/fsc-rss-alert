## Problem Statement
v1은 금융위(FSC) 보도자료 RSS 1개를 폴링해 Slack/Telegram으로 푸시하는 "지인 1명용,
무료 영원히" 도구였다. v2는 맥락이 바뀐다: **컴플라이언스/AML 팀의 편의 도구**로, 7개 규제/제재 소스를 한곳에서
감시한다(거래소 맥락). 단, 규제가 강제하는 SLA 도구가 아니다 — 팀의 **수동 모니터링이
백스톱으로 남는다.** 그래도 "조용히 죽어서 아무도 모르는" 수집기는 편의 도구로서 무가치하므로,
**죽음 탐지·가시성**이 핵심 신뢰성 투자다. 반대로 컴플라이언스 등급 이중화·장기 감사보존은
불필요. "맥이 켜져 있어야 갱신된다"는 v1 방식은 팀 도구로서도 결함이다.

감시 대상 7개 소스:
1. 금감원(FSS) 보도자료
2. 금융위(FSC) 입법예고/규정변경예고
3. FIU(KOFIU) 제재공시
4. 금융위 법령해석
5. 금융위 비조치의견서
6. 외교부(MOFA) 독자제재
7. OFAC SDN (recent actions)

## Requirements
| ID | 요구사항 | 충족 방식 |
|----|---------|-----------|
| R1 | RSS/소스 유연 등록 | 소스 레지스트리(config) + 메커니즘별 어댑터 |
| R2 | 소스별 Slack 채널 1:1 라우팅 | bot이 채널 자동생성(`feed-{slug}`) + `chat.postMessage` 라우팅 |
| R3 | 소스별 수집 전략 (4~5 메커니즘) | RSS/HTML/JSON(board·datatables)/벌크 어댑터 |
| R4 | 누락·중복 없는 상태관리 | sqlite, 소스별 seen 격리, 첫실행 기준선 |
| R5 | EC2 상주 (맥 의존 제거) | t3.micro + cron + docker (Approach A) |
| R6 | main 머지 → Docker → 자동 배포 | GitHub Actions(OIDC)→ECR, EC2 cron이 `:latest` pull |
| R7 | 발송 로그 (운영용 경량) | `audit_log` 테이블 (AuditPort), 90일 prune. 컴플라이언스 등급 보존 불필요 |
| R8 | 죽은 소스 탐지 + 운영 가시성 | 즉시 실패 알림 + daily digest(소스별 마지막 성공시각) |

## What Makes This (the right) Approach
- **핵심 발견이 설계를 정함:** "RSS 피드 등록"이라는 요청 프레이밍이 틀렸다. 조사 결과
  7개 중 네이티브 RSS는 사실상 1개(MOFA, 그것도 쿠키 우회 필요)뿐이다. 나머지는 RSS가
  없거나(FSS/FIU/better.fsc), 입법예고는 RSS 범위 밖이거나(FSC), RSS가 2025-01-31에
  폐지됐다(OFAC). 실제로는 **여러 수집 메커니즘**으로 나뉜다.
- 따라서 이건 "RSS 등록기"가 아니라 **소스별 어댑터를 가진 범용 수집기**다. 이 재프레이밍이
  헥사고날(ports & adapters) 아키텍처를 자연스럽게 부른다 — 교과서적 적용 사례.
- 좋은 소식: 어떤 소스도 Cloudflare/captcha/헤드리스 브라우저가 필요 없다. 전부 평범한
  HTTP 클라이언트로 도달 가능. 마찰은 둘뿐(MOFA 쿠키 307, OFAC 서명된 S3 리다이렉트).

## 소스 인벤토리 (조사 확정)
| # | 소스 | 메커니즘 | 엔드포인트 / 방식 | 함정 |
|---|------|---------|------------------|------|
| 1 | FSS 보도자료 | HTML 스크래핑 | `list.do?menuNo=200218`, 서버렌더 284KB | detail: `view.do?nttId=` |
| 2 | FSC 입법예고 | HTML 스크래핑 | `po040301` 서버렌더 | RSS는 보도자료용일 뿐 입법예고엔 없음 |
| 3 | FIU 제재공시 | JSON (eGov board) | `POST /cmn/board/selectBoardListFile.do` | boardConfig 복제 + 페이징 |
| 4 | 법령해석 | JSON (DataTables) | `POST selectReplyCaseLawreqList.do` | `draw/start/length/columns[]` 파라미터, 인증 불필요 |
| 5 | 비조치의견서 | JSON (DataTables) | `POST selectReplyCaseOpinionList.do` | 위와 동일 패턴 |
| 6 | MOFA 독자제재 | 네이티브 RSS | `rss.do?brdId=235` → application/rss+xml | `TMOSHCooKie` 307 self-redirect → 쿠키자 + 2-hit |
| 7 | OFAC SDN | 벌크 파일 diff | `sanctionslistservice.ofac.treas.gov/.../SDN.XML` | ~28MB, 302→서명 S3, 다운로드 후 diff |

**메커니즘 수 정정(리뷰 반영):** JSON 소스는 응답뿐 아니라 **요청 모양도 다르다**.
FIU의 eGov board(boardConfig + 페이징)와 better.fsc의 DataTables(`draw/start/length/columns[]`)는
요청 구성이 근본적으로 달라서, `JsonApiAdapter` 하나로 셋을 다 흡수하기보다 **`JsonBoardAdapter`와
`DataTablesAdapter`로 분리**하는 게 정직하다. 즉 메커니즘은 4개가 아니라 **5개**
(RSS / HTML / JSON-board / DataTables / 벌크)로 본다. 그래도 핵심 절감은 유효하다: better.fsc
법령해석·비조치 2소스가 `DataTablesAdapter` 하나를 공유한다.

## Constraints
- **Free tier 우선:** AWS free tier(EC2 t3.micro 12개월 750h/월 무료, ~1GB RAM) 범위 운영.
- **신뢰성(편의 도구 등급):** 중복 없음, 죽으면 알아챌 것(즉시 + 매일 digest). 누락은 수동
  백스톱이 있어 치명적이진 않으나 최소화. 컴플라이언스 등급 이중화·장기보존은 불필요.
- **헥사고날을 얇게:** 7소스 폴러에 엔터프라이즈 레이어링(서비스레이어, DI프레임워크,
  추상팩토리)은 과설계. ports = Protocol, adapters = 클래스. 그 이상 쪼개지 않는다.
- **코어 순수성(contents-hub 차용):** 도메인 코어는 httpx/bs4/feedparser/slack-sdk를 절대
  import하지 않는다. I/O 라이브러리는 전부 어댑터 뒤. 코어 단위테스트가 빠르고 Item 파이프라인이 순수.

## Premises (이 세션에서 합의됨)
1. **재프레이밍 수용** — RSS 등록기가 아니라 메커니즘별 어댑터를 가진 수집기. 소스 추가 =
   어댑터 선택 + config 한 블록.
2. **컴플라이언스 등급 상태관리** — sqlite. 소스별 seen_ids 격리 + 마지막 성공 시각 +
   발송 감사 로그.
3. **EC2 상주** — GitHub Actions 상주는 부적합. v1에서 Actions 러너가 한국 정부망
   443 timeout(= GitHub IP 차단)을 겪어 맥 launchd로 옮겼던 이력. EC2(가능하면 서울
   리전)가 이 문제를 통째로 우회.
4. **Slack: bot token + `chat.postMessage` + 채널 자동생성** — webhook-per-source 폐기.
   봇이 소스별 채널을 `feed-{slug}` 규칙으로 자동 생성하고 라우팅한다.
5. **"조용한 실패" 탐지가 1급 요구사항** — HTML 스크래핑·JSON 소스는 사이트가
   마크업/파라미터를 바꾸면 에러 없이 0건을 반환하며 죽는다. 즉시 실패 알림 + daily digest로 탐지.

## Prior Art (yansfil/contents-hub 비교·차용)
같은 모양의 문제(이질적 소스 → 정규화 Item → 상태 → 챗 전달)를 푸는 개인용 콘텐츠 수집기를
참고했다. 도메인은 다르나(개인 read-it-later vs 팀 컴플라이언스), 데이터 모델링 규율을 차용:
- **구조화 실패 분류(`FetchFailureReason` + `infer_from_error`)** → 우리 premise #5 구체화(위 Reliability).
- **content-hash dedup 폴백** → 안정 키 없는 항목 대비(위 Dedup 전략).
- **파서 버전 핀** → 한국 정부 HTML 레이아웃 변경 대응.
- **코어 순수성(provider SDK 코어 미import)** → 우리 헥사고날과 정렬(위 Constraints).
- **선언형 소스 레지스트리 + execution_method dispatch** → 우리 YAML 레지스트리 방향 독립 검증.

차용하지 않음(의도적): **Lens(주제 기반 채널 fan-out)** — 우리 요구는 1소스:1채널이라 지금은 과설계.
**마크다운 이중 아카이브** — `audit_log` + export로 갈음(R7 옵션).
우리가 우위: 배포 성숙도(Docker/EC2/Terraform/cron), 공식 JSON-API·벌크파일 경로, 직접 per-channel 전송.

## Approaches Considered (런타임 토폴로지)
헥사고날 코어는 확정. 갈린 건 인바운드 어댑터(스케줄러)와 배포 모양뿐이었다.

### Approach A: cron + CLI (서버 없음) — 선택됨
- cron이 N분마다 poll 컨테이너를 실행하고 프로세스는 죽는다. daily digest는 두 번째 cron.
  소스별 주기는 config `interval` + sqlite `last_run`으로 게이트. 상태는 마운트된 sqlite.
- Effort: M. Risk: Low. Reuses: 기존 `app.py:run()` 루프, `Notifier` ABC, `state.py`.
- Pros: 매 실행이 격리 프로세스 → `docker run ... poll` 한 줄로 손 재현. 크래시가 상주
  프로세스를 오염 못 시킴(self-heal). free tier 메모리 여유. R8 digest는 cron 한 줄.
- Cons: 실시간 `/health` 없음(daily digest로 갈음). EC2 단일(SPOF) → heartbeat로 보완.
  동시성 주의 필요(아래 Concurrency 참조).

### Approach B: 상주 워커 + APScheduler + FastAPI /health
- 프로세스 상주, 내부 스케줄러가 소스별 cadence로 폴링. `/health` 실시간, 향후 Slack
  자가등록 webhook 수신까지 확장. Effort: M-L. Risk: Med.
- Cons: 상주 프로세스 관리(누수·데드락은 restart로 못 잡음). v1엔 과할 가능성. 자가등록은
  소스마다 전용 어댑터가 필요해 `/addfeed <url>` 한 줄 추가가 불가 → B의 핵심 명분이 약하다.

### Approach C: 관리형 서버리스 (EventBridge/Lambda) — 폐기
- Lambda 컴퓨트는 free tier 가능하나 (a) sqlite 못 쓰니 상태 외부화 필요, (b) 한국
  정부사이트 IP 도달성이 Lambda에서 될지 미검증. free tier에서 이득 없이 리스크만 증가.

## Recommended Approach
**Approach A (cron + CLI) + 헥사고날 코어.** 컴플라이언스가 요구하는 건 추론·디버깅 가능한
신뢰성이지 멀티-AZ HA가 아니다. 단일 EC2 + docker + cron + sqlite가 가장 디버깅하기 쉽고,
"매번 죽고 새로 깨어남"이 장애에 더 강하다. 헥사고날 덕에 나중에 실시간 health나 자가등록이
필요해지면 **인바운드 어댑터만** APScheduler+FastAPI(B)로 교체하면 되고 코어·소스·알림·상태
어댑터는 한 줄도 안 건드린다.

## Architecture
```
   inbound                  DOMAIN CORE (순수, I/O 없음)        outbound ports
   ─────────                ──────────────────────────         ──────────────
   CLI + cron  ──poll(src)─▶ first-run? → fetch → dedup ──▶ SourcePort
   (스케줄러)                 → new? → notify → audit          ├ RssAdapter (feedparser + fetch profile)
   ─ poll                     → advance(seen)                  ├ HtmlScrapeAdapter (FSS, FSC입법예고)
   ─ digest (R8)                                               ├ JsonBoardAdapter (FIU)
                                                               ├ DataTablesAdapter (법령해석, 비조치)
                                                               └ BulkDiffAdapter (OFAC SDN)
                                                            ──▶ StatePort → SqliteStateRepo
                                                            ──▶ NotifierPort → SlackBotNotifier (이미 ABC 존재)
                                                            ──▶ AuditPort → SqliteAuditLog
```

**기존 파일 이동:**
- `feed.py` → `adapters/rss.py` + `adapters/http_fetch.py`. `RssAdapter`는 RSS 파싱만 알고,
  사이트별 HTTP 마찰은 fetch profile이 처리한다. MOFA의 쿠키자+307 2-hit은 `MofaCookieGateFetcher`
  책임이고, feedparser는 바이트만 파싱한다. (v1의 `fetch_feed` + `parse_entries` 분리를 계승하되
  소스별 cookie name은 RSS 어댑터에 넣지 않는다)
- `app.py:run()`의 diff/신규감지/첫실행/실패처리 로직 → `core/poll.py`(순수) + `core/dedup.py`.
  **첫 실행 분기와 oldest-first 정렬은 반드시 보존**(v1 `app.py:116-121`, `127`).
- `notify.py`의 `Notifier` ABC → `ports.py`의 `NotifierPort`; `SlackNotifier` → bot token 방식.
  "FSC RSS 폴링 실패" 같은 고정 문자열은 소스명 주입형으로 리네임.
- `state.py` → `adapters/state_sqlite.py`. Telegram(`TelegramNotifier`)은 **옵션 폴백으로 유지**(제거 아님).
- 하드코딩 `config.FEED_URL` → 소스 레지스트리(YAML config).

**다형성 전략 (소스 수만큼 풀구현 회피):** 공통 도메인 타입 `Item`으로 모든 어댑터가 정규화 →
다운스트림(dedup/diff/notify/audit/state)은 한 번만 작성하고 전 소스에 동작. 어댑터는
**메커니즘별 베이스 5개 + 소스별 config/fetch profile**. Template Method로 베이스가 fetch/페이징/재시도를
처리하고, 사이트별 쿠키·리다이렉트 우회는 별도 fetch profile로 둔다. 소스별로는
**`build_request()`와 `map_row()`만** override(보통 각 ~10줄). config-only
YAML-DSL은 함정(Scrapy 열화판)이라 회피.

```python
# ports.py
class SourcePort(Protocol):
    def fetch(self) -> list[Item]: ...

# domain.py — published를 datetime으로 정규화(소스별 KST 포맷 흡수, 정렬·digest용)
@dataclass(frozen=True)
class Item:
    item_id: str          # 중복제거 키. 안정 키 우선(RSS guid/link, JSON idx, OFAC SDN UID),
    title: str            # 없으면 content-hash 폴백(아래 Dedup key 전략 참조)
    link: str
    published: datetime | None

# core/poll.py — 어댑터 종류와 무관, 한 번만 작성
def poll(source: SourceCfg, adapter: SourcePort, state, notifier, audit):
    if not due(source, state):          # last_run + interval 게이트 (poll이 레지스트리 순회)
        return
    items = adapter.fetch()             # 어떤 메커니즘이든 list[Item]
    if state.is_first_run(source.id):   # 첫 실행: 기준선만 저장, 알림 0 (v1 보존)
        state.advance(source.id, items)
        return
    new = dedup(source.id, items, state)        # 소스별 seen 격리
    for it in oldest_first(new):                # v1 ordering 보존
        notifier.send(source.channel_id, it)
        audit.log(source.id, it)
        state.mark_seen(source.id, it.item_id)  # 발송 성공분만 seen에 커밋
```

```python
class DataTablesAdapter(SourcePort):
    def build_request(self, cfg):       # DataTables 전용 요청 구성(공유)
        return {"draw": 1, "start": 0, "length": 20, **cfg.params}
    def map_row(self, r, cfg) -> Item:  # 소스별 override(~10줄)
        return Item(item_id=str(r["lawreqIdx"]), title=r["title"],
                    link=cfg.detail_url.format(id=r["lawreqIdx"]),
                    published=parse_kst(r.get("regDt")))
```

## State Schema (sqlite)
- `sources(id, name, mechanism, parser_version, channel_id, interval_minutes, last_run_at, last_success_at, consecutive_failures, failure_alert_sent)`
- `seen_items(source_id, item_id, first_seen_at, PRIMARY KEY(source_id, item_id))` — 소스별 격리,
  소스당 최근 N개로 prune. **단일 seen 모델만 사용**(v1의 capped-list가 아니라 이 테이블이 진실원).
- `audit_log(id, source_id, item_id, title, channel_id, slack_ts, sent_at, status)` — 발송 감사.
  `slack_ts`(Slack 메시지 타임스탬프)로 재발송 dedup·향후 편집/스레드 가능. 보존기간은 Open Q.

### Dedup key 전략 (contents-hub 차용)
`item_id`를 우선순위로 결정: ① 안정 키(RSS `guid`/`link`, JSON `idx`, OFAC SDN `UID`)
→ ② 안정 키 없으면 `content://{source_id}/<sha256(title|body|published)>` content-hash 폴백.
`UNIQUE(source_id, item_id)`로 **DB 레이어에서** 중복 강제(앱 로직 의존 아님).

### Concurrency (리뷰 반영 — 중요)
cron이 매 tick 새 컨테이너를 띄우므로 동시 실행 위험이 있다. OFAC(28MB)처럼 느린 폴이
interval을 넘기면 두 컨테이너가 같은 sqlite를 쓴다 → 중복 발송/락 경합.
- **전역 `flock`**: poll 진입 시 단일 파일락 획득, 실패하면 즉시 종료(직전 tick이 아직 도는 중).
- **sqlite WAL 모드** + busy_timeout 설정.
- **sqlite는 컨테이너가 아니라 EC2 호스트 디스크에** 둔다(예: `/data/feed.db`). cron이 매번
  `docker run --rm -v /data:/data ...`로 컨테이너를 새로 띄우고 버리므로, db를 컨테이너 안에 두면
  사라진다. 호스트 디렉토리를 bind-mount해서 db가 컨테이너 재생성에도 살아남게 한다. (호스트
  루트 디스크가 이미 EBS다. 스냅샷 백업을 쉽게 하려면 `/data`만 별도 EBS 볼륨으로 분리하는 건 선택.)
  **항상 한 번에 한 컨테이너만** 동작(flock) 보장.

## Reliability & Operations
- **구조화된 실패 분류(contents-hub 차용 — premise #5의 구체화):** fetch/파싱 예외를
  `FetchFailureReason` enum으로 분류하고 `infer_from_error()`가 예외 텍스트를 매핑한다 —
  `STRUCTURE_CHANGED`/`BLOCKED`/`LOGIN_REQUIRED`/`NOT_FOUND`/`TIMEOUT`/`NETWORK`/`UNKNOWN`.
  알림 강도를 사유별로 차등: `STRUCTURE_CHANGED`·`BLOCKED`·`NOT_FOUND`(=놓침 위험)는 **즉시
  `feed-ops` 알림**; `TIMEOUT`·`NETWORK`(=일시)는 재시도 후 임계값에서만. 정상인데 0건이 평소
  주기보다 길게 지속돼도 죽음 의심으로 digest에 표시.
- **즉시 실패 알림 유지(v1 계승):** 소스별 `consecutive_failures`가 임계값 도달 시 `feed-ops`에
  1회 알림(`failure_alert_sent`로 중복 방지). 24시간 digest만 기다리지 않는다.
- **politeness/차단 회피:** 소스별 최소 interval 바닥값, v1의 retry/backoff(`fetch_retries`,
  `retry_delay`) 계승, 브라우저형 `User-Agent` 유지(특히 한국 정부 사이트).
- **fetch/파싱/발송 실패 시 seen 전진 금지**(v1 계승) → 누락 방지.
- **파서 버전 핀(contents-hub 차용):** HTML/JSON 소스 config에 `parser_version`. `STRUCTURE_CHANGED`가
  뜨면 파서를 새 버전으로 올리고, 버전 명시로 롤백 가능.

## Resolved (이전 Open Q에서 확정)
- **daily digest = KST 09:00**, 운영 채널 `feed-ops`.
- **audit_log 보존 = 90일 prune** (편의 도구라 경량). 컴플라이언스 등급 장기보존 불필요.
- **전달 순서 = 수직 슬라이스(plan-ceo-review).** 슬라이스1 = MOFA RSS + better.fsc 법령해석을
  끝→끝(어댑터→sqlite→slack→EC2→cron→digest)으로 먼저 검증. 이후 5소스는 config+어댑터 반복.
- **OFAC 폴링 주기 = 12시간 고정.** 28MB를 자주 받지 않는다. diff는 직전 SDN **UID 집합**을
  sqlite에 보관하고 신규 designation만 추출. 파싱은 **스트리밍(iterparse)**으로 t3.micro 메모리
  내 처리(전체 28MB를 메모리에 적재하지 않음). `Item.item_id` = SDN UID.
- **Slack 채널 자동 생성 채택.** 스코프 = `chat:write` + `channels:manage` + `channels:read`.
  봇이 소스별 public 채널을 만들고 자기 입장 + 메시지 발송까지 자동. **사람(컴플라이언스 팀원)
  초대는 사람이** 한다(봇은 생성+입장까지만). 채널은 public(워크스페이스 내 발견성).

### Slack 채널 네이밍 (확정)
규칙: **`feed-{기관}-{주제}`**, 소문자/하이픈/ASCII. config의 각 소스에 `slug` 필드 → 채널명 = `feed-{slug}`.
Slack API 채널명은 ASCII 제약이 있으므로 실제 채널 handle은 ASCII로 유지한다. 대신 사람이 읽는 표시는
한글을 우선한다: config의 `name`을 한국어 표시명으로 두고, Slack 메시지 본문/digest/채널 topic·purpose에는
`name`을 사용한다. 즉 `#feed-fsc-lawreq`처럼 자동화 친화적인 handle을 만들되, 채널 설명과 메시지에는
`금융위 법령해석`처럼 사람이 바로 읽는 이름을 노출한다.

| 소스 | 채널명 |
|---|---|
| FSS 보도자료 | `feed-fss-press` |
| FSC 입법예고 | `feed-fsc-legislation` |
| FIU 제재공시 | `feed-fiu-sanctions` |
| 금융위 법령해석 | `feed-fsc-lawreq` |
| 비조치의견서 | `feed-fsc-opinion` |
| MOFA 독자제재 | `feed-mofa-sanctions` |
| OFAC SDN | `feed-ofac-sdn` |
| 운영(전일 digest + 실패알림) | `feed-ops` |

봇 로직(멱등): 채널 없으면 `conversations.create` → 받은 `channel_id`를 sqlite에 저장. 이미 있으면
(`name_taken`) `conversations.list`로 기존 채널 조회해 재사용. channel_id가 진실원이므로
이름이 바뀌어도 발송은 id로 한다.

## Distribution Plan (CI/CD)
- **레지스트리 = AWS ECR (private).** AWS 올인 맥락이라 네이티브. free tier: private 12개월
  500MB/월 저장 무료(작은 파이썬 이미지면 충분, 오래된 태그 prune). same-region ECR→EC2 pull은
  데이터전송 무료·고속.
- **빌드/push:** main 머지 → GitHub Actions가 이미지 빌드 → **OIDC 역할**(장기 액세스키 없이)로
  ECR에 push (`:latest` + `:<sha>`).
- **배포(pull 기반, push 트리거 불필요):** EC2 cron이 poll 직전 `docker pull ...:latest`를 한 번
  실행하고 `docker run`. 인증은 **EC2 인스턴스 역할(ECR 읽기전용)** + **ECR credential helper**로
  자동(수동 `docker login`·12h 토큰 갱신 없음). **SSM·watchtower 불필요.**
- 런타임이 cron 배치 컨테이너이므로 무중단 배포 부담 없음(다음 tick이 새 이미지 사용).
- 시크릿(Slack bot token 등)은 EC2의 `.env`(chmod 600) 또는 SSM Parameter Store.

## Infrastructure as Code (Terraform)
AWS 리소스는 전부 Terraform으로 관리한다. 콘솔 클릭 금지(재현·리뷰·롤백 가능).
- **관리 대상:** ECR 리포지토리(lifecycle policy로 태그 prune), EC2 인스턴스 + 보안그룹(아웃바운드만,
  인바운드 SSH는 최소/SSM 세션매니저 선택), IAM 인스턴스 프로파일(ECR read) + OIDC provider + Actions
  push 역할, 선택적 EBS 볼륨 + 어태치.
- **state 관리:** Terraform state는 S3 백엔드 + DynamoDB 락(또는 Terraform Cloud). 로컬 state 금지.
- **시크릿 제외:** Slack 토큰 등은 Terraform state에 넣지 않는다(SSM Parameter Store에 수동/별도 관리,
  Terraform은 파라미터 "이름"만 참조).
- **EC2 부트스트랩:** user_data로 docker + ECR credential helper 설치 + cron 등록(또는 Ansible/cloud-init).
  cron이 `docker pull :latest` 후 `docker run`.
