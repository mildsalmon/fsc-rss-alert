## Problem Statement
국내에 거주하며 금융 관련 업무를 하는 지인이, 금융위원회(FSC) 보도자료를
매번 사이트(https://www.fsc.go.kr/ut060101)에 직접 들어가 수동으로 확인하고 있다.
핵심 고통은 "매번 수동 확인"의 번거로움. 원하는 건 새 보도자료가 올라오면
**비동기로 감지되어 Slack 또는 모바일 앱으로 알림(push)이 오는 것**.
리더를 여는 pull 방식이 아니라, 알림이 날아오는 push 방식이어야 한다.

## What Makes This Cool
- 기성품(Feedly 등)으로 깔끔하게 안 풀리는 빈틈을 정확히 메운다 (아래 Premises 참고).
- 작고 무료지만 핵심 루프(가져오기 → 신규 감지 → 알림)를 직접 구현 = 폴링/diff/
  웹훅/크론/배포를 한 바퀴 학습할 수 있다.
- 이미 기다리는 실제 사용자 1명이 있다. 대부분의 사이드 프로젝트엔 없는 자산.
- 확장하면 "한국 정부 정보를 push로 받는 도구"라는 더 넓은 빈틈으로 이어진다.

## Constraints
- 최대한 가볍게. 불필요한 기능 전부 제외, 핵심만.
- 무료로 계속 돌아야 한다 (지인 1명을 위해 유료 결제는 비현실).
- 재미 + 학습 + 지인 돕기를 동시에 만족 ("여러 마리 토끼").

## Premises (이 세션에서 검증됨)
1. **실제 첫 사용자의 요구 = push 알림.** "새 금융위 보도자료 → Slack/모바일 알림".
   pull 리더가 아님. (확인됨)
2. **기성품으로 안 풀린다.**
   - Feedly는 새 글 push 알림을 네이티브로 지원하지 않음 (IFTTT/Zapier 필요).
   - Feedly의 Slack 연동은 Pro도 아닌 Threat/Market Intelligence(기업용 고가) 전용.
   - Slack의 네이티브 RSS 앱은 FSC 피드에서 실패함 (아래 4번).
   (Feedly 공식 문서로 확인)
3. **원래 "한국 정부 RSS가 해외 IP를 차단한다"는 가설은 (적어도 FSC에 대해) 틀렸다.**
   - 미국 IP(Anthropic WebFetch)와 국내 IP(curl) 모두 https 피드를 정상 수신.
   - User-Agent도 변수가 아니었음 (Slackbot/Telegram/빈 UA/브라우저 전부 동일 동작).
   - http:// 주소는 302로 https://로 리다이렉트되고, https는 200 + text/xml + 208KB
     정상 피드를 반환.
4. **그러나 "외부 서비스가 한국 정부 피드를 안정적으로 못 삼킨다"는 현상은 진짜다.**
   - Slack에 https 피드 등록 시 "SSL 인증서를 확인할 수 없습니다" 에러로 실패 (사용자 재현).
   - 인증서 체인 자체는 완결돼 있고(GlobalSign, 국내 맥에서 verify OK), 다른 해외
     클라이언트(WebFetch)는 성공 → 원인은 Slack fetcher의 엄격한 TLS 검증 또는
     접속 노드/인증서 차이. 어느 쪽이든 **사용자가 손댈 수 없는 FSC 서버 영역**.
   - 결론: 내가 통제하는 봇이 피드를 폴링(어떤 fetcher로도 성공)하고, 결과를
     Slack 웹훅/텔레그램으로 push하면 이 문제가 통째로 우회된다.
5. **확장 여지(나중).** RSS가 아예 없거나 깨진 다른 한국 정부 사이트로 넓히기.
   현재 v1 범위에서는 제외.

## Approaches Considered
### Approach A: GitHub Actions 크론 + 웹훅 (선택됨)
- 요약: GitHub Actions 스케줄(예: 20분)이 작은 스크립트 실행 → FSC https 피드 fetch
  → 직전 본 항목과 비교 → 신규만 Slack 웹훅/텔레그램으로 POST. 상태는 repo 커밋 또는
  Actions 캐시.
- Effort: S (인간 ~반나절 / CC ~15분). Risk: Low.
- Reuses: GitHub Actions 무료 크론, Slack Incoming Webhook / Telegram Bot API.
- Pros: 서버 0, 비용 0, 이번 주에 end-to-end 동작. 핵심 루프 학습에 최적.
- Cons: Actions 크론 몇 분~십몇 분 지연 가능. repo 60일 무활동 시 워크플로 비활성화
  (매 실행 커밋으로 활성 유지 가능). 실시간 대화형 자가등록엔 부적합.

### Approach B: 상시 워커 + DB (이상적 구조, 현 시점 과설계)
- 요약: Fly.io/Railway 무료 또는 $5 VPS에 상주 서비스. 피드/구독자/채널을 설정으로
  관리, 상태 SQLite, 소스·알림 어댑터 플러그인.
- Effort: M-L (인간 ~3-5일 / CC ~1-2시간). Risk: Med.
- Pros: 다중 피드·사용자·채널, 장기 확장 최고.
- Cons: 지인 1명 + 피드 1개엔 명백한 과설계. 서버 유지보수 필요.

### Approach C: Cloudflare Workers + Cron Triggers + KV (가벼운 정석)
- 요약: 엣지 Worker가 Cron Trigger로 폴링 + KV로 상태 저장 + 텔레그램 웹훅 수신까지.
  지인이 봇에게 `/add <url>` 보내면 즉시 자가등록 가능.
- Effort: S-M (인간 ~1일 / CC ~35분). Risk: Low-Med.
- Pros: 서버 없이 안정적, 상태 깨끗(KV), 자가등록이 1급 기능, 다중 피드 확장 쉬움.
- Cons: wrangler 첫 셋업 한 겹 더. JS/TS.

## Recommended Approach
**Approach A**, 단 **v1은 FSC 피드 1개를 하드코딩**하고 자가등록은 v1.5로 미룬다.
이유: "가볍게 + 재미·학습 + 곧 쓸 지인"이라는 목표에 가장 정확히 맞고, 검증된
고통(FSC 알림)을 이번 주에 끝에서 끝까지 동작시킬 수 있다. 쓸모가 확인되고
자가등록을 "친구가 봇에게 메시지로 추가"하는 매끄러운 형태로 원하게 되면 그때 C로 이주.

## Implementation Decisions (리뷰 반영, 구현 전 확정)
- **검증된 피드 엔드포인트 (literal):** `https://www.fsc.go.kr/about/fsc_bbs_rss/?fid=0111`
  - http:// 주소는 302로 위 https로 리다이렉트됨. 처음부터 https를 직접 쓴다.
  - 이 엔드포인트가 200 + `text/xml` + ~208KB 정상 피드(최신 10건)를 반환함을 확인.
- **중복제거 키:** RSS `<guid>` 사용. 없으면 `<link>`로 폴백. pubDate는 키로 쓰지 않음
  (동시각·재발행 시 중복/누락 위험). 단일 값이 아니라 **최근 본 ID 약 50개를 set으로 저장**
  → 순서 뒤바뀜·크론 스킵에도 self-heal.
- **첫 실행:** 기준선(현재 항목 ID들)만 저장하고 알림은 쏘지 않음 (폭탄 알림 방지).
- **다중 신규 항목:** 한 번에 여러 건이면 **오래된 것부터(oldest-first)** 전송. 채널 rate
  limit 초과 우려 시 throttle.
- **실패 처리:** fetch 5xx/timeout 또는 XML 파싱 실패 시 **last-seen 상태를 전진/커밋하지
  않음**(누락 방지). 알림 POST 실패도 동일. 연속 N회 실패 시 본인에게 1회 자가 알림.
- **상태 저장 = 확정:** repo에 last-seen ID 파일 커밋. 단 **상태가 바뀔 때만 커밋**
  (매 실행 커밋 금지 → 히스토리 비대 방지). 실제 새 글이 충분히 자주 커밋을 만들어
  60일 비활성화도 막음. 필요 시 주 1회 keep-alive 커밋 추가.
- **워크플로 권한:** state 커밋을 위해 `permissions: contents: write` + 기본 `GITHUB_TOKEN`.
- **스택:** Python(feedparser가 RSS·리다이렉트·TLS 한 방에 처리) 권장. Node도 무방.

## Open Questions
- 알림 채널 v1 기본값: Slack Incoming Webhook(업무 맥락) vs Telegram Bot(모바일 push가
  가장 간단). 지인이 실제로 어디서 알림을 받을지로 결정.
- 폴링 주기: 20~30분이면 보도자료 성격상 충분한가? (지인 확인)

## Success Criteria
- 지인이 FSC 사이트에 들어가지 않고도 새 보도자료를 알림으로 받는다.
- 같은 글이 두 번 오지 않고(중복 없음), 새 글을 놓치지 않는다(누락 없음).
- 비용 0으로 계속 돈다.
- (학습) 폴링·diff·웹훅·크론·배포 루프를 직접 구현해봤다.

## Distribution Plan
개인/지인용 도구. "배포 = GitHub 저장소 생성 + 워크플로 활성화 + Slack 웹훅 URL
(또는 Telegram 봇 토큰)을 Actions Secret에 등록". GitHub Actions가 곧 런타임이자
CI/CD이므로 별도 배포 파이프라인 불필요.

## Next Steps (concrete build tasks)
1. GitHub repo 생성, Slack Incoming Webhook(또는 Telegram 봇) 1개 발급.
2. 스크립트 v0: `https://www.fsc.go.kr/about/fsc_bbs_rss/?fid=0111` fetch → 파싱 →
   콘솔에 최신 항목 출력. (네트워크/파싱부터 동작 확인)
3. 신규 감지: `<guid>`(폴백 `<link>`) 기반, 최근 ID set(~50개) 저장/비교. 첫 실행은
   "기준선만 저장, 알림 안 쏨". fetch/파싱 실패 시 상태 전진·커밋 안 함.
4. 알림 전송: 신규 항목을 oldest-first로 Slack 웹훅/Telegram에 POST. 제목 + 링크 + 날짜.
5. GitHub Actions 크론(20~30분) + Secret(웹훅 URL) + `permissions: contents: write`.
   상태가 바뀔 때만 커밋.
6. 지인에게 채널 연결해주고 며칠 관찰: 중복/누락/지연 체크.

## What I noticed about how you think
- 당신은 "한국 정부 RSS가 해외 IP를 차단하는 것 같다"는 가설을 **먼저 검증하고
  만들자**고 했다. 제품을 짓기 전에 핵심 가정을 의심하는 건 드문 본능이다.
- "결국 지인이 필요하다고 해서 만들려는거거든" — 추상적 시장이 아니라 실제 한 사람의
  실제 요청에서 출발했다. 그리고 그 사람의 진짜 고통("매번 수동 확인")을 정확히 집었다.
- "사실 feedly로 받아도 문제 없을 것 같긴 해" — 자기 아이디어를 죽일 수도 있는 사실을
  스스로 꺼냈다. 대부분은 이걸 숨기거나 외면한다. 이 정직함이 헛제품을 막았고,
  결국 Feedly가 push를 못 한다는 걸 확인하며 진짜 빈틈을 찾아냈다.
- "Slack이나 모바일 app으로든 알람이 오면 되는거라서 feedly에서 알람 지원이 되는지부터
  확인해봐야겠다" — 만들기 전에 status quo부터 점검하는 태도가 일관됐다.
- 끝까지 "가볍게"를 놓지 않으면서도 자가등록 같은 확장을 떠올렸고, 그걸 v1에 욱여넣지
  않고 v1.5로 미루는 절제를 보였다.
